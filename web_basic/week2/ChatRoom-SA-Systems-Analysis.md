## 系統概述

### 1.1 專案背景
WebSocket 聊天室是一個基於即時雙向通訊技術的 Web 應用程式，允許多個使用者同時線上聊天。此系統展示了 WebSocket 技術在即時通訊應用中的實際運用。

### 1.2 專案目標
- 實現即時多人聊天功能
- 展示 WebSocket 雙向通訊技術
- 提供簡潔易用的使用者介面
- 支援多使用者同時線上互動

### 1.3 專案範圍
- **包含範圍**：使用者登入、即時訊息傳送、多人聊天、連線管理
- **不包含範圍**：使用者註冊、訊息歷史儲存、檔案傳輸、私人聊天

## 需求分析

### 2.1 利害關係人識別

| 利害關係人 | 角色說明 | 主要需求 |
|------------|----------|----------|
| 一般使用者 | 聊天室參與者 | 能夠登入並即時聊天 |
| 系統管理員 | 系統維護者 | 監控系統運行狀態 |
| 開發團隊 | 系統開發者 | 技術實現與維護 |

### 2.2 業務流程分析

#### 2.2.1 使用者登入流程
```
使用者輸入名稱 → 點擊登入 → 建立WebSocket連線 → 發送登入訊息 → 顯示登入成功
```

#### 2.2.2 訊息傳送流程
```
使用者輸入訊息 → 點擊送出 → WebSocket傳送至伺服器 → 廣播給所有連線用戶 → 顯示在聊天視窗
```

#### 2.2.3 使用者離線流程
```
使用者關閉網頁/斷線 → 觸發onClose事件 → 從連線清單移除 → 清理相關資源
```

## 功能性需求

### 3.1 核心功能需求

#### FR001 - 使用者登入功能
- **描述**：使用者可以輸入暱稱並登入聊天室
- **輸入**：使用者暱稱（必填，非空白）
- **處理**：驗證輸入，建立WebSocket連線
- **輸出**：登入成功/失敗訊息，登入通知廣播

#### FR002 - 即時訊息傳送
- **描述**：已登入使用者可以傳送訊息給所有線上使用者
- **輸入**：文字訊息內容
- **處理**：透過WebSocket廣播訊息
- **輸出**：訊息顯示在所有使用者的聊天視窗

#### FR003 - 訊息接收顯示
- **描述**：即時接收並顯示其他使用者的訊息
- **輸入**：來自伺服器的訊息資料
- **處理**：解析JSON格式訊息
- **輸出**：格式化顯示「使用者名稱 說：訊息內容」

#### FR004 - 連線狀態管理
- **描述**：管理使用者的連線狀態
- **處理**：追蹤連線建立、維持、斷開
- **輸出**：更新連線清單，通知其他使用者

### 3.2 使用案例圖

```
使用者
├── 登入聊天室 (FR001)
├── 傳送訊息 (FR002)
├── 接收訊息 (FR003)
└── 離開聊天室 (FR004)
```

## 非功能性需求

### 4.1 效能需求
- **NFR001**：系統應支援至少50個同時連線使用者
- **NFR002**：訊息傳送延遲不超過1秒
- **NFR003**：系統回應時間不超過3秒

### 4.2 可用性需求
- **NFR004**：系統可用性達到99%
- **NFR005**：支援主流瀏覽器（Chrome, Firefox, Safari, Edge）
- **NFR006**：提供友善的錯誤訊息

### 4.3 安全性需求
- **NFR007**：防止XSS攻擊（輸入驗證）
- **NFR008**：限制訊息長度防止濫用
- **NFR009**：基本的連線驗證

### 4.4 擴展性需求
- **NFR010**：架構支援水平擴展
- **NFR011**：易於增加新功能模組

## 技術架構分析

### 8.1 技術選型分析

| 技術類別 | 選用技術 | 選用原因 | 替代方案 |
|----------|----------|----------|----------|
| 後端框架 | Java WebSocket API | 標準API，穩定可靠 | Spring WebSocket |

| 應用伺服器 | Apache Tomcat | 輕量級，易部署 | Jetty, Undertow |
| 資料格式 | JSON | 輕量級，易解析 | XML, Protocol Buffers |

## 風險分析

### 9.1 技術風險

#### 9.1.1 高風險項目
1. **Session 管理問題**
   - **風險**：連線未正確清理導致記憶體洩漏
   - **影響**：系統效能下降，最終當機
   - **緩解**：實作完整的 @OnClose 處理

2. **WebSocket 連線穩定性**
   - **風險**：網路不穩定導致連線中斷
   - **影響**：使用者體驗不佳
   - **緩解**：實作重連機制和錯誤處理

#### 9.1.2 中等風險項目
1. **瀏覽器相容性**
   - **風險**：不同瀏覽器對 WebSocket 支援程度不同
   - **影響**：部分使用者無法使用
   - **緩解**：使用 SockJS 作為回退方案

2. **訊息格式錯誤**
   - **風險**：JSON 格式錯誤導致解析失敗
   - **影響**：訊息顯示異常
   - **緩解**：強化輸入驗證和錯誤處理

### 9.2 業務風險

#### 9.2.1 使用者體驗風險
- **風險**：大量使用者同時連線導致效能下降
- **緩解**：設定連線數限制和負載監控

#### 9.2.2 安全風險
- **風險**：惡意使用者發送大量訊息
- **緩解**：實作訊息頻率限制和內容過濾

## 系統限制

### 10.1 技術限制

1. **連線數限制**
   - 單一伺服器實例連線數受記憶體限制
   - 建議最大同時連線數：100-500（視伺服器規格）

2. **訊息儲存限制**
   - 無持久化儲存，重啟後訊息遺失
   - 無訊息歷史功能

3. **擴展性限制**
   - 使用靜態 ArrayList 儲存連線
   - 無法進行水平擴展（多伺服器部署）

### 10.2 功能限制

1. **使用者管理限制**
   - 無使用者認證機制
   - 無使用者權限管理
   - 暱稱可能重複

2. **聊天功能限制**
   - 僅支援文字訊息
   - 無私人聊天功能
   - 無檔案傳輸功能

### 10.3 建議改進方向

1. **短期改進**
   - 增加輸入驗證
   - 實作重連機制
   - 優化錯誤處理

2. **中期改進**
   - 整合資料庫儲存訊息歷史
   - 實作使用者認證
   - 增加私人聊天功能

3. **長期改進**
   - 使用 Redis 實作分散式 Session 管理
   - 實作負載平衡和水平擴展
   - 增加檔案傳輸和多媒體支援

## 結論

WebSocket 聊天室系統是一個展示即時通訊技術的良好範例。透過系統分析，我們識別出系統的核心功能、技術架構和潛在風險。雖然當前實作有一些限制，但提供了堅實的基礎架構，可以根據業務需求進行擴展和改進。

建議在實際部署前，優先解決高風險項目，並根據使用者回饋持續優化系統功能和使用者體驗。
